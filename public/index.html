Build High-Performance Async API Servers in a Blaze



What is Rammer?

Rammer is a framework dedicated to build high performance Async API servers on top of non-blocking (asynchronous) Ruby web server called Goliath. Rammer APIs are designed on top of REST-like API micro-framework Grape. Rammer is modular and integrates a powerful CLI called Viber to plug in and out its modules.

Getting Started

Rammer is available as a gem, to use it just install the gem:

gem install rammer

Usage:

rammer application_name 
cd application_name 
bundle install
This will install the missing gems and now application is ready to run the Goliath server.

To start the server:

ruby server.rb -vs
This command will run the Goliath server in the default port:9000

Running Tests

Rammer uses a code coverage analysis tool, called SimpleCov. Its available as a gem in ruby, giving you a complete code coverage for your work.

You can Run tests in command prompt by 

rake test

When running rake test, this automatically creates a index.html under folder coverage in root directory with details of rammer code coverage. Loading this into any web browser shows the complete code usage of rammer generators.

Rammer Modules

Currently Rammer supports 3 modules which can be accessed using Viber.

  • Authentication  : Enables user registration
  • Authorization   : Enables user authorization
  • Oauth           : Enables Oauth service provider

    • Viber - The Magic Hook!

    Viber is a powerfull CLI tool which is used to plugin or unplug the rammer modules. Its usage mounts the specified module onto the application, generating the required files.

    Usage:

    $ viber module < COMMAND > < MODULE_NAME >

    < COMMAND >

  • -p or -plugin (Plugs in the specified module)
  • -u or -unplug (Unplugs the specified module)

    • < MODULE_NAME >

  • authentication
  • authorization
  • oauth
    • This will generate the required files

    Endpoints Mounted with Rammer Modules

    Authentication Module Endpoint

                  
     Basic path : /v1/authentication
    
              
                
                
    
                    Resource 
                    Description
                
    
                    
                
                    
    
                        /sign_up                                    
                        Enables user registration using basic credentials : email, MD5 hash password and a redirect url of base applicat-ion. Allows auto login funcationality returning a session token for further api access.
                    
    
                    
              
    
            
                  
    Request format: 
    
              
    {
    
              
    "email"=> "User email",
    
              
    "password" => "MD5 hash encrypted password",
    
              
    "redirect_uri" => "Url to which the token should be redirected"
    
              
    }

    Authorization Module Endpoints

                  
     Basic path : /v1/authorization
    
              
                
                
    
                    Resource 
                    Description
                
    
                    
                
                    
    
                        /sign_in                                
                        Enables user login using basic credentials : email, MD5 hash password and a redirect url of base application. Al-lows user authorization returning a session token for further api access.
                    
    
                    
    
                        /sign_out                               
                        Enables user logout using basic credentials : email and session token allotted during sign_up/sign_in. The gener-ated session token is invalidated for further access.
                    
    
                    
              
    
            
                  
    Request format for /sign_in: 
    
              
    {
    
              
    "email"=> "User email",
    
              
    "password" => "MD5 hash encrypted password",
    
              
    "redirect_uri" => "Url to which the token should be redirected"
    
              
    }
    
              
    Request format for /sign_out: 
    
              
    {
    
              
    "email"=> "User email",
    
              
    "session_token" => "Session token obtained during sign in",
    
              
    "redirect_uri" => "Url to which the token should be redirected"
    
              
    }

    Oauth Module Endpoints

                  
     Basic path : /v1/oauth
    
              
                
                
    
                    Resource 
                    Description
                
    
                    
                
                    
    
                        /register                                        
                        Enables client registeration using basic credentials: client name and a redirect url of base application.Returns client access details with client id, secret hash key and basic authorization code to be further used for oauth api access.

                    
    
                    
    
                        /authorize                
                        Enables user authorization checking functionality for oauth api access. Functionality can be accessed by specifying an authorization page url.

                    
    
                    
    
                        /request_token                                        
                        Enables user oauth authorization using credentials: client_id (obtained during client registration), redirect_uri(specified while registration), response_type, request_token, username (session token obtained during user authorization). Retu-rns request_token for further api access.

                    
    
                    
    
                        /access_token                                         
                        Enables user oauth authorization using credentials: client_id (obtained during client registration), redirect_uri(specified while registration), response_type, request_token, username (session token obtained during user authorization). Retu-rns an access token for api access.

                    
    
                    
    
                        /token                    
                        Enables user authorization checking functionality for oauth api access. Functionality can be accessed by specify-ing an authorization page url.

                    
    
                    
    
                        /invalidate_token         
                        Enables user authorization checking functionality for oauth api access. Functionality can be accessed by specify-ing an authorization page url.

                    
    
                    
                    
              
    
            
                  
    Request format for /register: 
    
              
    {
    
              
    "name"=> "Client's name",
    
              
    "redirect_uri" => "URL to which the oauth should be redirected",
    
              
    }
    
              
    Request format for /authorize: 
    
              
    {
    
              
    "client_id"=> "Client's registered ID",
    
              
    "username" => "Authorized user's session id",
    
              
    "redirect_uri" => "URL to which the oauth should be redirected",
    
              
    "response_type" => "token" (Keyword to return request token)
    
              
    }
    
              
    Request format for /request_token: 
    
              
    {
    
              
    "response_type":"code",
    
              
    "client_id":"client id received on registration",
    
              
    "redirect_uri":"redirect uri specified during registration",
    
              
    "username":"session token value obtained"
    
              
    }
    
              
    Request format for /access_token: 
    
              
    {
    
              
    "response_type":"token",
    
              
    "client_id":"client id received on registration",
    
              
    "redirect_uri":"redirect uri specified during registration",
    
              
    "username":"session token value obtained",
    
              
    "request_token" : "request token obtained from /request_token endpoint."
    
              
    }
    
              
    Request format for /token: 
    
              
    {
    
              
    "client_id"=> "Client's registered ID",
    
              
    "authorization" => "Basic authorization key generated while client registration",
    
              
    "host_name" => "Third party client's name",
    
              
    "redirect_uri" => "URL to which the oauth should be redirected",
    
              
    "response_type" => "token" (Keyword to return request token)
    
              
    }
    
              
    Request format for /invalidate_token: 
    
              
    {
    
              
    "client_id"=> "Client's registered ID",
    
              
    "authorization" => "Basic authorization key generated while client registration",
    
              
    "host_name" => "Third party client's name",
    
              
    "redirect_uri" => "URL to which the oauth should be redirected",
    
              
    "response_type" => "token" (Keyword to return request token)
    
              
    }
